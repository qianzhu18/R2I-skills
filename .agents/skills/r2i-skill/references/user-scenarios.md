# User / Scenario / Pain Map

Use this file to adapt depth, tone, and emphasis before producing the final output.

## Primary Users

### 1. Job seeker or career switcher

Typical scenario:
- Wants to turn a public repo, side project, internship repo, or open-source contribution into interview material and resume bullets

Main pains:
- Can describe files, but cannot explain architecture or tradeoffs
- Writes vague bullets that sound fake under follow-up questioning
- Does not know what the strongest interview talking points are
- Needs someone to ask realistic follow-up questions before using the material externally

Output emphasis:
- business value and ownership framing
- defensible STAR bullets
- 1-minute and 3-minute speaking scripts
- likely follow-up questions and weak spots

### 2. Student or junior developer

Typical scenario:
- Wants to actually understand a strong repository and use it as a learning project

Main pains:
- Overwhelmed by repo size and unfamiliar patterns
- Does not know where to start reading
- Understands isolated files but not system flow
- Does not know whether they truly understand the project until someone challenges them

Output emphasis:
- reading order
- key concepts and architecture map
- practice tasks with increasing difficulty
- plain-language explanations before jargon

### 3. Working developer preparing for interviews, promotion, or internal transfer

Typical scenario:
- Needs to recover and package prior work into a clear project story

Main pains:
- Remembers having built features, but not the strongest evidence or framing
- Struggles to translate code and implementation detail into leadership or impact language
- Needs honest but high-signal summaries quickly
- Wants to pressure-test claims before using them in interviews or promotion docs

Output emphasis:
- system boundaries and tradeoffs
- evidence-backed ownership framing
- role-specific interview questions
- concise bullets and contribution narratives

## Scenario Heuristics

If the user asks to "analyze a repo" with no clear goal:
- assume they want `full-loop`, but keep the first response compact

If the user asks for "how to learn" or "how to read this repo":
- prioritize learning order and practice tasks

If the user asks for "how to explain in an interview":
- prioritize architecture, tradeoffs, and questions that an interviewer would ask next

If the user asks for "resume", "STAR", or "bullet points":
- prioritize evidence-backed bullets and honesty checks

## Pain Categories To Solve

Always try to reduce at least these four pains:
- Understanding gap: the user cannot see the system-level picture
- Translation gap: the user cannot turn code work into interview language
- Evidence gap: the user lacks defensible proof for strong claims
- Focus gap: the user does not know what to study or say first
- Pressure-test gap: the user has not yet faced the follow-up questions that expose weak understanding
